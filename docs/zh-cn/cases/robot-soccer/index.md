# 机器人足球案例

案例定位：

| 维度 | 内容 |
| --- | --- |
| 类型 | 应用场景案例 |
| 对应示例 | `examples/robot-soccer/` |
| Failure mode | `blocked_shot` |
| 学习重点 | 动作前提和通道检查如何进入规则系统 |

## 背景

机器人足球策略往往不是从零开始：系统已经有视觉模型、状态估计、动作接口和大量手工规则。真正痛点通常是规则维护，而不是完全没有策略。

这个案例把问题压缩成一个教学场景：机器人已经拿到球，球门就在前方，但射门通道被对手挡住。学习目标不是“赢下一整场比赛”，而是让策略在动作前检查关键前提，并把这个前提固化成可测试规则。

## 当前系统假设

| 模块 | 可能现状 | HL 关注点 |
| --- | --- | --- |
| 感知 | YOLO/检测器识别球、机器人、场线 | 状态表示是否稳定 |
| 策略 | 大量 if/else rule | 规则是否可测试、可回放 |
| 反馈 | 比赛结果、丢球、碰撞、回放 | 反馈能否被编码智能体读取 |
| 更新 | 人手改规则 | 智能体能否补测试后改规则 |

## 最小课程实验

课程实验使用二维足球策略环境，代码位于：

```text
examples/robot-soccer/
```

运行：

```bash
npm run examples:robot-soccer
npm run examples:robot-soccer:feedback
npm run examples:test
```

这个实验只保留一个关键问题：直接射门通道被对手挡住时，策略是否应该先换路线。

课程环境：

- 状态：球位置、我方位置、对方位置、球门方向。
- 动作：追球、带球、射门、回防。
- 反馈：是否射门成功、是否丢球、是否卡住。
- 更新对象：策略规则、边界检测、回归测试。

## 当前实验结果

| 策略 | 结果 | 解释 |
| --- | --- | --- |
| baseline | `blocked_shot` | 拿到球后过早射门，没有检查射门通道 |
| heuristic | `goal` | 先识别 blocked lane，换路线后再射门 |

## 验证标准

- 每条新规则必须有对应失败场景。
- 每轮更新必须保留旧场景回归测试。
- 不以单场胜利作为成功标准，而以“多场景不退化”作为课程目标。
