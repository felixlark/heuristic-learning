# 交通模拟案例

这个案例来自飞书任务线索：把翁家翌的启发式学习用于武汉东湖交通模拟器。当前它不是公开论文结论，而是一个待验证的应用方向，所以本页先记录问题结构与课程化拆解方式。

## 案例拆解

| 问题 | 记录 |
| --- | --- |
| 环境 | 城市/景区交通模拟器，当前线索指向武汉东湖 |
| 状态 | 路网拓扑、车流密度、拥堵位置、信号灯相位、事件/管制信息 |
| 动作 | 调整信号灯、推荐分流、改变限速、触发管制策略 |
| 反馈 | 平均通行时间、排队长度、拥堵消散速度、规则冲突、人工复盘 |
| 更新对象 | 交通规则库、信号灯策略、异常检测器、仿真测试场景 |
| 验证 | 固定场景 replay、峰值流量 stress test、旧场景回归测试 |

## 为什么适合 HL

交通模拟里的策略通常不是单一神经网络可以直接替代的对象。它会混合规则、人工经验、仿真配置、实时指标和局部例外。HL 的切入点不是“直接训练一个万能策略”，而是让编码智能体在失败场景中维护这些结构：

1. 找到拥堵没有消散、分流互相冲突或信号灯策略失效的场景。
2. 把场景转成 replay/probe。
3. 修改规则、阈值或状态检测器。
4. 重新运行仿真与回归测试。
5. 把本轮规则变化写回实验记录。

## 最小实验草图

本仓库已经把它落成一个纯 Python `traffic-grid` 示例：

```bash
npm run examples:traffic-grid
npm run examples:traffic-grid:feedback
npm run examples:test
```

对应文件：

```text
examples/traffic-grid/
experiments/traffic-grid/latest.json
tests/test_traffic_grid.py
```

这个实验包含：

- 路口节点：入口、主干道、景区停车场、出口。
- 状态：每条边的车辆数与等待时间。
- baseline：只按当前最短队列放行。
- heuristic：加入事件管制、下游拥堵保护和公平性约束。
- probe：单纯最短队列会把车导向已经拥堵的下游路段。

当前实现把问题进一步压缩成 downstream spillback probe：baseline 只看上游最大队列，heuristic 先保护下游容量。它的目标不是还原真实东湖路网，而是先形成可测试的 HL 闭环。真实仿真接入时，应保留同一组 probe，避免规则从课程版迁移到真实环境时丢失验证标准。
