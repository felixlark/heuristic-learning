# 从 RL/DL 到 HL

HL 不是 Deep Learning 或 Reinforcement Learning 的替代品。它更像是在“可被代码表达、可被测试反馈驱动”的问题上，提供另一种学习对象。

| 维度 | Deep Learning | Deep RL | Heuristic Learning |
| --- | --- | --- | --- |
| 更新对象 | 神经网络参数 | 策略/价值网络参数 | 软件结构、规则、测试、记忆 |
| 更新机制 | 反向传播 | 奖励驱动的梯度/采样更新 | 编码智能体直接修改代码 |
| 反馈来源 | 数据集损失 | 环境奖励、轨迹 | 测试、日志、回放、奖励、人类反馈 |
| 优势 | 表征能力强 | 可处理复杂决策 | 可解释、可审查、可快速固化 |
| 风险 | 数据/算力成本 | 样本效率低、难调 | 规则冲突、工程型遗忘 |

## 为什么参考 Karpathy 的仓库

Karpathy 的 `llm.c` 给这个仓库一个重要启发：好教学仓库不只是“解释概念”，还要把核心机制压缩到读者能运行、能改、能理解的代码里。

HL 仓库也采用这个标准：

- 理论页面要能指向代码。
- 代码示例要短到可以读完。
- 每个示例都要有命令和测试。
- 结果要能被复现，而不是只停留在叙述。

## 为什么参考 EasyVibe

EasyVibe 的价值在于课程组织：学习路线、阶段任务、图文说明、实践入口和附录可以自然地服务不同水平读者。HL 也需要这种结构，因为它目前仍处于思想沉淀期，读者需要被带着走完“概念 -> 案例 -> 实验 -> 研究问题”的路径。

## 继续形成研究问题

更具体的问题定义、度量维度和实验范式见 [学习闭环](/zh-cn/theory/learning-loop) 与 [研究框架](/zh-cn/theory/research-framework)。本仓库后续新增案例时，应优先补齐 baseline failure、heuristic patch、feedback report 和 regression test，而不是只补叙述。
